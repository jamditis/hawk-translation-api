# Autocontext skill design

A Claude Code plugin that accumulates project knowledge across sessions and developers through structured lessons, hooks, and a curator agent. Inspired by greyhaven-ai/autocontext's knowledge persistence patterns, built as a standalone skill.

## Problem

Claude Code sessions start cold. Every new session forgets:
- API patterns and gotchas (trailing slashes, auth headers, status sync behavior)
- Schema and constants (game data structures, database models, config shapes)
- Conventions and handoff protocols (cross-developer coordination, naming patterns)
- Performance characteristics (which paths are hot, what anti-patterns cause regressions)
- What previous sessions already tried and failed

This wastes time on repeat corrections, produces easily avoidable errors, and gets worse as codebases grow more complex. The problem is amplified when multiple developers (each with their own Claude instances) work on the same codebase.

## Solution

A `.autocontext/` directory in each project root, checked into git, containing structured lessons that hooks load, enforce, and update automatically. No external services, no API keys, no daemons.

## Knowledge model

### Lesson format

`lessons.json` is a JSON array of lesson objects:

```json
[
  {
    "id": "lesson_a1b2c3d4",
    "schema_version": 1,
    "category": "efficiency",
    "text": "POST /api/ideas/ requires trailing slash — without it FastAPI returns 405",
    "context": "Dashboard Board API, FastAPI route configuration",
    "confidence": 0.9,
    "validated_count": 3,
    "last_validated": "2026-03-13T22:00:00Z",
    "created": "2026-03-10T14:00:00Z",
    "created_by": "joe@officejawn",
    "supersedes": null,
    "deleted": false,
    "tags": ["api", "fastapi", "dashboard", "machine:officejawn"]
  }
]
```

Key fields:
- **schema_version** — integer, incremented on format changes. Enables future migration.
- **confidence** — 0.0-1.0. See confidence lifecycle below.
- **validated_count** — how many sessions confirmed this lesson still holds.
- **supersedes** — lesson ID this one replaces. Old lesson stays in the array with `deleted: true`.
- **deleted** — tombstone flag. Deleted lessons remain in the array (with `deleted: true`) so the merge driver doesn't re-add them from another developer's copy. Pruned during `/autocontext-review`.
- **tags** — plain strings for file paths, module names, concepts. Tags prefixed with `machine:` are filtered on non-matching machines (e.g., `machine:officejawn` only appears on officejawn sessions).

### Lesson categories

| Category | What accumulates | When it helps |
|----------|-----------------|---------------|
| `efficiency` | API patterns, conventions, gotchas, workflow rules | Prevents repeat mistakes during implementation |
| `codebase` | Schema, constants, architecture decisions, data flow | Reduces cold-start time, prevents misunderstanding the system |
| `optimization` | Performance findings, anti-patterns, benchmarks, timing baselines | Catches regressions, guides optimization work |

### Confidence lifecycle

Uses time-based decay (not session-count-based, which would require a global counter synced across machines):

- New auto-extracted lesson: confidence 0.5
- Each session that uses the lesson and it proves correct: +0.1 (cap at 1.0)
- Each session where the lesson was wrong or irrelevant: -0.2
- Not validated in 60 days: decay by 0.1 per 30 days (based on `last_validated` timestamp vs current date)
- Below 0.3: filtered from SessionStart injection (still in file, available via `/autocontext-review`)
- Superseded: kept in array with `deleted: true`, moved to `archive/superseded.json` during next `/autocontext-review`

### Playbook generation

`playbook.md` is auto-generated from `lessons.json` and should never be hand-edited. Format:

```markdown
# Project playbook (auto-generated)
Last updated: 2026-03-13T22:00:00Z | 47 active lessons

## Efficiency (14 lessons)
- **[0.9]** POST /api/ideas/ requires trailing slash (api, fastapi)
- **[0.85]** X-API-Key header required for remote requests (api, auth)
...

## Codebase (28 lessons)
- **[0.95]** Belt items use distance-offset struct, not per-item NetworkObjects (game-engine)
...

## Optimization (5 lessons)
- **[0.7]** Graffiti wall sticker resize causes layout thrash — batch DOM updates (rendering)
...
```

Generated by `scripts/generate-playbook.py` (a standalone Python script with no dependencies beyond stdlib). Called by the SessionEnd hook after lesson persistence and by `/autocontext-review` after curation.

### Directory structure

```
.autocontext/
├── lessons.json          # JSON array of all lessons (including tombstoned deletions)
├── playbook.md           # Human-readable summary (auto-generated, never hand-edit)
├── config.yaml           # Project-specific settings (checked into git)
├── config.local.yaml     # Machine-specific overrides (gitignored)
├── .gitattributes        # Custom merge driver for lessons.json
├── .gitignore            # Ignores config.local.yaml and cache/
├── cache/                # Transient session state (gitignored)
│   └── session-lessons.json  # Parsed + filtered lessons cached at session start
└── archive/
    └── superseded.json   # Pruned lessons (audit trail)
```

**Gitignored files:** `config.local.yaml`, `cache/`. Everything else is committed and shared.

### Config

Two config files — shared and local:

```yaml
# .autocontext/config.yaml (checked into git — shared settings)
project_name: my-project
max_session_lessons: 15          # Max lessons injected at session start
confidence_threshold: 0.3        # Min confidence to inject
staleness_days: 60               # Days before confidence decay starts
performance_baselines: true      # Enable PostToolUse performance tracking

builtin_rules:
  tautological_test_check: true
  no_mock_everything: true
  no_happy_path_only: true
  no_assert_true: true
  test_independence: true
```

```yaml
# .autocontext/config.local.yaml (gitignored — per-machine settings)
identity: joe@officejawn
# Overrides any shared config values for this machine
```

**Identity resolution order:**
1. `config.local.yaml` → `identity` (if set)
2. `~/.claude/autocontext.yaml` → `identity` (global default from `/autocontext-setup`)
3. Fallback: `$(git config user.email)@$(hostname)`

## Hook architecture

Lesson extraction uses a three-phase approach to avoid the `SessionEnd` timeout constraint (default 1.5s, capped by `CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS`). No heavy LLM work happens at session end.

```
During session:  UserPromptSubmit → detect correction patterns → accumulate candidates
At session end:  SessionEnd → write candidates + bump validated_count (pure file I/O, <1.5s)
Next session:    SessionStart → curate pending candidates via claude -p → load lessons
```

### SessionStart — load knowledge + curate pending lessons

**Trigger:** SessionStart (command hook)
**Token cost:** Low normally; medium if pending lessons exist from previous session
**Timeout:** 60s (needs headroom for optional `claude -p` curation)

**Phase 1: Curate pending lessons from previous session**

1. Check for `.autocontext/cache/pending-lessons.json`. If it exists, a previous session flagged lesson candidates.
2. Call `claude -p` with the curator prompt (see below) and the pending candidates. This runs at session start, where there's no tight timeout.
3. For each curated lesson:
   - If approved: add to `lessons.json` with confidence 0.5
   - If rejected: discard
   - If duplicate of existing: skip
4. Delete `pending-lessons.json` after processing.
5. Regenerate `playbook.md` via `scripts/generate-playbook.py`.

If the user chose "always ask before persisting" in `/autocontext-setup`, the curated lessons are shown via `/autocontext-review` instead of auto-persisting.

If `claude -p` fails or times out, leave `pending-lessons.json` for the next session to retry.

**Phase 2: Load lessons for this session**

1. Check for `.autocontext/lessons.json` in project root. If missing or malformed JSON, warn and skip (don't crash the session).
2. Filter: `confidence >= confidence_threshold`, `deleted != true`, machine tags match current host
3. Rank by relevance:
   - Lessons whose tags match recently changed files rank highest. Uses `git diff --name-only HEAD~5 2>/dev/null` — if this fails (fresh repo, detached HEAD, fewer than 5 commits), falls back to `git diff --name-only HEAD 2>/dev/null`, then skips relevance ranking entirely. This is a heuristic — branch switches and rebases may produce noisy results, which is acceptable.
   - Then by confidence (descending)
   - Then by recency (most recently validated first)
4. Take top `max_session_lessons` lessons
5. Write filtered set to `.autocontext/cache/session-lessons.json` (cached for PreToolUse to read without re-parsing)
6. Check if merge driver `autocontext-union` is configured in local git config. If not, warn.
7. Format as a context block injected into the session

Output format:
```
[autocontext] Loaded 12 lessons for my-project (3 efficiency, 7 codebase, 2 optimization)
Top lessons for current work area:
- [0.9] POST /api/ideas/ requires trailing slash — without it returns 405
- [0.85] DeepL handles 7 languages, Google handles ht/hi/ur — different backends
- [0.8] Glossary terms are in DB (terms_json), not a static file
```

### PreToolUse — catch known mistakes and enforce test quality

**Trigger:** PreToolUse (prompt-based hook, filtered to `Edit`, `Write`, `Bash` only)
**Token cost:** Medium (runs on Edit/Write/Bash tool calls only — not Read/Glob/Grep/Agent)

This is a **prompt-based hook**, not a command hook. The hook prompt has access to the pending tool call and can reason about whether lessons apply. Command hooks can't do the semantic matching needed here.

**Scope restriction:** Only fires on `Edit`, `Write`, and `Bash` tool calls. `Read`, `Glob`, `Grep`, and `Agent` calls are excluded — they don't modify state and don't benefit from warnings. This keeps token cost proportional to actual changes, not read-heavy exploration.

Hook prompt:

```
You have access to the following project lessons from .autocontext/cache/session-lessons.json.
When the user is about to edit, write, or run a command, check if any lessons are relevant
to the file or command being touched. If so, inject a brief warning.

For test files (matching *test*, *spec*, *_test.*, *.test.*), also apply these advisory checks:
- Are assertions based on desired behavior (from the task/spec), not current implementation output?
- Is there at least one error/edge case?
- Would this test fail if the feature broke?
- Are mocked return values being used as the expected assertions?

Only inject warnings when genuinely relevant. Do not warn on every tool call.
Cap at 3 warnings per tool call.
```

The prompt-based hook has inherent token cost since it runs through the LLM. To manage this:
- Only fires on Edit/Write/Bash (not read-only tools)
- The session-lessons cache (written by SessionStart) keeps the lesson set small
- The hook prompt itself is short — the lessons are the bulk
- Users who find the cost too high can disable it in `config.yaml` (`pretooluse_hook: false`)

### PostToolUse — track performance + deterministic test checks

**Trigger:** PostToolUse (command hook)
**Token cost:** Low (only fires when command/file matches patterns)

Two responsibilities:

**Performance tracking** (fires on Bash commands matching test/build patterns: `pytest`, `npm test`, `npm run test`, `npm run build`, `cargo test`, `go test`, `vitest`, `jest`, `make test`):

1. Parse output for timing information and pass/fail counts
2. Compare against baselines stored in `config.yaml` under `baselines:`
3. If tests got slower (>10% regression) or new failures appeared, inject a warning
4. Update baselines with new timing data

Only active when `performance_baselines: true` in config.

**Deterministic test quality checks** (fires on Edit/Write of test files):

After a test file is written or edited, run fast regex checks on the file contents:

| Rule | Detection method |
|------|-----------------|
| `no_assert_true` | Regex scan for `assert True`, `assert result is not None`, `assert len(result) > 0`, `self.assertTrue(True)` |
| `no_happy_path_only` | Count test methods — if all method names start with `test_success` / `test_valid` / `test_happy` and none contain `error` / `fail` / `invalid` / `edge`, flag (advisory — may produce false positives) |

These run on the *resulting* file contents (PostToolUse has the written file), not on the intended input (PreToolUse). This gives accurate detection. No LLM cost.

### UserPromptSubmit — detect lesson candidates

**Trigger:** UserPromptSubmit (command hook)
**Token cost:** Negligible (fast pattern matching on user message text)

Lightweight correction detection that runs after each user message. No LLM call — just pattern matching.

Scans user message text for correction patterns:
- "no, use X instead" / "that's wrong" / "don't do that" / "actually it should be"
- "remember that" / "remember this" / "keep in mind"
- "the correct way is" / "it needs to be" / "you forgot that"

When a pattern matches:
1. Extract the user message and the preceding assistant response (for context)
2. Append a candidate entry to `.autocontext/cache/pending-lessons.json`:
   ```json
   {
     "user_message": "no, POST /api/ideas/ needs a trailing slash",
     "preceding_context": "I'll call POST /api/ideas without...",
     "files_touched": ["api/routes.py"],
     "timestamp": "2026-03-13T22:15:00Z"
   }
   ```
3. No LLM reasoning at this stage — just capture the raw signal

Candidates are curated by `claude -p` at the *next* session's SessionStart (see above).

### SessionEnd — persist session metadata

**Trigger:** SessionEnd (command hook)
**Token cost:** Negligible (pure file I/O)
**Timeout:** Fits within default 1.5s — no LLM calls

Pure file operations only:

1. Bump `validated_count` and `last_validated` on any lessons from `session-lessons.json` that were loaded this session and not contradicted
2. Write session metadata to `pending-lessons.json` (append, not overwrite):
   - Session timestamp, identity, files touched
   - Any correction candidates accumulated by UserPromptSubmit
3. Regenerate `playbook.md` via `scripts/generate-playbook.py`

No `claude -p` call. No transcript analysis. This fits comfortably in the 1.5s SessionEnd timeout.

## Curator prompt

Used by the SessionEnd hook's `claude -p` subprocess call:

```
You are a knowledge curator. Given a session transcript excerpt, extract
lessons worth persisting for future Claude Code sessions on this project.

A good lesson:
- Is specific to this project (not general programming knowledge)
- Is actionable (tells a future session what to do or avoid)
- Survived the session (the fix worked, the pattern held)
- Would save time if known at session start

Reject if:
- General knowledge any developer would know
- Too vague to act on ("be careful with X" without specifics)
- Contradicts existing knowledge without strong evidence
- About a one-time task that won't recur

For each lesson, output JSON:
{"lessons": [
  {
    "category": "efficiency|codebase|optimization",
    "text": "concise, actionable description",
    "context": "where in the codebase this applies",
    "tags": ["file-paths", "module-names", "concepts"]
  }
]}

If no lessons are worth extracting, output: {"lessons": []}
```

## Built-in test quality rules

Two types: **deterministic** (pattern-matching, run as command hooks) and **advisory** (need LLM reasoning, run as part of the PreToolUse prompt-based hook).

### Deterministic rules (command hook patterns)

| Rule | Detection method |
|------|-----------------|
| `no_assert_true` | Regex scan for `assert True`, `assert result is not None`, `assert len(result) > 0`, `self.assertTrue(True)` |
| `no_happy_path_only` | Count test methods — if all method names start with `test_success` / `test_valid` / `test_happy` and none contain `error` / `fail` / `invalid` / `edge`, flag |

These can run as fast regex checks in a command hook with no LLM cost.

### Advisory rules (prompt-based, part of PreToolUse)

| Rule | What the LLM checks |
|------|---------------------|
| `tautological_test_check` | Are assertions derived from reading the implementation rather than from the task/spec? |
| `no_mock_everything` | Are mocked return values the same as the expected assertions? |
| `test_independence` | Would this test still pass if the feature under test didn't exist? |

These require reasoning about intent and are included in the PreToolUse prompt-based hook. They carry the associated token cost and can produce false positives. Users can disable individual rules in `config.yaml`.

## Cross-developer sharing

### Transport: git

`.autocontext/` is committed to the repo (except `config.local.yaml` and `cache/`). Both developers pull the same lessons. No special sync infrastructure.

### Merge conflicts

`lessons.json` uses a custom git merge driver registered in `.autocontext/.gitattributes`:

```
lessons.json merge=autocontext-union
```

The merge driver (`scripts/merge-driver.sh`):
1. Parse both versions as JSON arrays
2. Union by lesson ID (both sides' new lessons get kept)
3. For lessons modified on both sides: use additive merge for counters (`validated_count` = sum of both increments from common ancestor), take most recent `last_validated`, take higher `confidence`
4. Preserve tombstones (`deleted: true`) from either side — if either side deleted, it stays deleted
5. Write merged result

**Setup requirement:** Git merge drivers must be configured in each developer's local `.git/config` — `.gitattributes` only declares which files use the driver. `/autocontext-init` installs the driver automatically. If a developer clones without running init, the SessionStart hook detects the missing merge driver and warns:

```
[autocontext] Warning: merge driver 'autocontext-union' not configured.
Run /autocontext-init to set up cross-developer sharing, or lessons.json
merges may produce broken JSON.
```

As a safety net: if `lessons.json` fails to parse (from a bad merge or corruption), the SessionStart hook warns and skips lesson injection rather than crashing the session.

### Identity

Identity comes from `config.local.yaml` (gitignored), the global `~/.claude/autocontext.yaml`, or falls back to `git config user.email` + hostname. Never stored in the shared `config.yaml`.

### Deletion propagation

Deleted lessons use a tombstone pattern: `"deleted": true` stays in the array. The merge driver preserves tombstones from both sides, preventing a deleted lesson from being re-added by another developer's stale copy.

**Tombstone pruning:** Tombstones are never auto-pruned. They accumulate until manually removed via `/autocontext-review`. This is safer than trying to detect "both devs synced" (which isn't reliably knowable from git state). The review command shows tombstones with a "permanently remove" option. For most projects, tombstone accumulation is negligible — a few deleted lessons per month adds bytes, not kilobytes.

## Slash commands

### /autocontext-setup

First-run plugin configuration. Runs once after install using AskUserQuestion for guided setup:

```
Step 1: "What should we call you in lesson attribution?"
  → Text input (e.g., "joe@officejawn", "kevin", "dev-1")
  → Stored in ~/.claude/autocontext.yaml as global identity

Step 2: "Which built-in test quality rules do you want enabled?"
  → Multi-select with AskUserQuestion:
    - Tautological test check (tests that describe code instead of testing behavior)
    - No mock everything (tests where mocks are the assertions)
    - No happy path only (require error/edge cases)
    - No bare assertions (flag assert True / assert is not None)
    - Test independence (flag tests that pass without their feature)

Step 3: "How aggressively should lessons be loaded at session start?"
  → Conservative: 5 lessons, confidence >= 0.7
  → Balanced: 15 lessons, confidence >= 0.3 (recommended)
  → Aggressive: 25 lessons, confidence >= 0.1

Step 4: "How should new lessons be persisted at session end?"
  → Auto-persist with curator validation (recommended)
  → Always ask before persisting
  → Auto-persist everything (no curator)
```

Writes `~/.claude/autocontext.yaml` with the selected defaults. These apply to all projects unless overridden per-project.

### /autocontext-init

Initialize `.autocontext/` in the current project. Uses AskUserQuestion for project-specific setup:

```
Step 1: (if CLAUDE.md exists) "This project has a CLAUDE.md with existing
         gotchas and patterns. Seed initial lessons from it?"
  → Yes, extract and let me review each one
  → Yes, extract automatically
  → No, start fresh

Step 2: "Will other developers use Claude Code on this repo?"
  → Yes, set up cross-developer sharing (git merge driver, identity tagging)
  → No, just me

Step 3: "Track test/build performance baselines?"
  → Yes (enables PostToolUse hook for test timing)
  → No
```

Then creates:
1. `.autocontext/` directory structure (including `.gitignore` for `config.local.yaml` and `cache/`)
2. `config.yaml` with answers (inherits global defaults from `~/.claude/autocontext.yaml`)
3. `config.local.yaml` with identity
4. Empty `lessons.json` (or seeded from CLAUDE.md)
5. `.autocontext/.gitattributes` with merge driver (if sharing enabled)
6. Installs merge driver in local git config (if sharing enabled)

### /autocontext-review

Interactive lesson curation using AskUserQuestion:
1. List all active lessons sorted by confidence
2. For each, show: text, confidence, validated count, created by, age
3. Options: approve (bump confidence), edit, delete (tombstone), supersede, skip
4. Show tombstoned lessons separately with option to permanently remove
5. Regenerate `playbook.md`

### /autocontext-status

Quick stats:
- Total lessons by category (active vs tombstoned)
- Average confidence
- Most-validated lessons (top 5)
- Stalest lessons (lowest confidence, oldest validation)
- Lessons by developer
- Lessons added this week
- Merge driver status (configured or not)

## Example use cases

The skill is project-agnostic. Here are examples of how different types of projects benefit:

### Translation/content pipelines
A translation API with AI quality scoring and human review. Knowledge accumulates about which content types get mis-scored, terminology patterns, and per-language quirks. Success metric: reduction in human reviewer corrections over time.

### Multi-developer game projects
A complex codebase where multiple developers each run their own Claude instances. Knowledge accumulates about schema shapes, authority models, build sequences, and handoff protocols. Lessons sync via git so both developers' Claude instances share the same understanding.

### Multi-service infrastructure
A system with bots, dashboards, schedulers, and CRM workflows. Knowledge accumulates about API patterns (auth headers, endpoint quirks), status sync rules, and processing workflows. Prevents repeated mistakes on service integration points.

### Any large codebase
Performance characteristics, architectural decisions, test patterns, and deployment gotchas. The optimization category tracks build/test timing baselines and flags regressions.

## Non-goals

- No external services or daemons
- No direct LLM API calls (uses `claude -p` subprocess for curator/analysis)
- No semantic search (tag-based matching is simpler and sufficient)
- No training or fine-tuning (knowledge is prompt-level, not weight-level)
- No MCP server (hooks are lighter and always available)
